Why?

In order to align multiple text blocks, possibly of varying sizes, along the horizontal axis using CSS; the correct distance to align things by has to be calculated. If not we end up with misalignment, which can be very unpleasant.

One tool I use to understand this distance is a Baseline Grid, its a visual aid used to enable text alignment. The Baseline Grid and Typographic Measuring System are inextricably linked; the latter is used to automate calculations which align text blocks on the former.

To squeeze even more use out of it we can make a single rem equate to any computed value we want by applying percentages to the <html> element. This way a single rem unit (the measurement used for incrementally traversing the baseline grid) will yield a computed pixel font-size that suits.

75% = 12px
87.5% = 14px
112.5% = 18px

If your base font-size is 12px, this means 1rem = 12px, this knowledge enables quite a lot of flexibility if you like things to be in multiples of 6.

Vertical harmony

Everything I’ve learnt recently about typography leads towards units of measurement being the fundamental building blocks of order and harmony in design. A Baseline Grid can be used as a visual aid to enable it by indicating what areas of a layout should accommodate text, and visa-versa; all in multiples of a number such as 6.

But surely we can already create this level of precision with CSS?

… yes, but it's far from straight forward. When we start to piece the various components of a web page together as part of a system, and want to align things vertically, and in a way that allows them to be re-assembled in varying order, so they align in a away that can be controlled (horizontally isn’t a problem), the current spec does not cater toward this mentality. One of the main reasons is there is no concept of a cap-height; there are several articles which explain the problem in great detail, here’s a good one, it refers to the problem as a “spooky box”.

Process

Pick a font-stack

I use Helvetica & system fonts quite a lot so didn't need to make too many adjustments to the default cap-height settings in Interior. If you pick different fonts you’ll need to work out what the cap-height is and apply it. if you plan to use Interior there’s a variable for cap-height and it’s tied into the baseline mixin.

Establish a unit for the baseline

I use 1rem (see above, link to HTML sizing part), as it works really well for several reasons—even though it has already been prescribed by Sassline (and others), I like to make life difficult for myself and went through the process of completely ignoring it by using unit-less values, in the end the way a browser renders unit-less values over unit values (e.g. rems), resulted in computed sizes that were incorrect [provide example, codepen] there were some severe limitations. Thankfully Sassline was available as a reference.

Make adjustments

This is the tricky—and possibly most time consuming—part, and what the visual baseline grid is for. To align text to “sit” on the baseline I need to see things in action, so I open up a browser window and view text on a baseline grid, if it doesn’t quite sit right I make adjustments until it does.

What I found of most use to get this correct was creating sample pages with varying degrees of content, paragraphs, headings, block-quotes, citations, images, figures with captions etc. then adding and removing things-each element impacts the next one down the baseline (unless you use an explicit grid to split up each element).
